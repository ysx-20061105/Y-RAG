# Research: 用户登录注册与认证（Sa-Token）

本文件用于支撑 `002-user-auth-satoken` 功能的设计与实现，记录关键技术决策、理由与备选方案，避免在实现阶段反复讨论基础问题。

## 决策 1：Sa-Token 在项目中的接入方式

**Decision**

- 使用 Sa-Token 作为唯一的登录态与会话管理框架，在 Spring Boot 中通过官方推荐的 Spring 集成方式接入：
  - 引入 Sa-Token Spring Boot Starter 依赖；
  - 在配置类中启用 Sa-Token（例如注册拦截器/过滤器）以对受保护接口做统一登录校验；
  - 在登录 / 登出逻辑中直接调用 Sa-Token 提供的工具方法（如 `StpUtil.login`、`StpUtil.logout`、`StpUtil.getLoginId` 等）完成会话创建与销毁；
  - 通过全局配置类统一设置 token 相关参数（过期时间、是否允许多端登录、token 风格等）。

**Rationale**

- 项目目标架构中已明确“认证使用 Sa-Token + JWT”，与现有技术栈（Java 21 + Spring Boot 3.x）兼容性好；
- Sa-Token 针对登录校验、权限控制、会话续期等提供了较完整的能力，可以减少自研 session 机制的工作量与安全风险；
- 通过在统一配置中集中管理 token 与会话策略，有利于后续扩展（如 RBAC、端端隔离、单点登录等）。

**Alternatives considered**

- 直接使用 Spring Security + 自定义 JWT：功能强大，但在本项目中会引入额外的配置与概念（过滤链、授权表达式等），与“Sa-Token 为主”的既定方向不符。
- 手写基于 `OncePerRequestFilter` + JWT 的认证逻辑：实现成本与安全风险都较高，且难以复用 Sa-Token 提供的会话与权限能力。

## 决策 2：密码存储与校验策略

**Decision**

- 用户密码不以明文存储，而是使用 BCrypt 算法进行单向哈希：
  - 在注册时对用户密码进行 BCrypt 哈希并存入数据库；
  - 在登录时，对用户输入的密码与数据库中的 BCrypt 哈希进行匹配校验；
  - 不在任意响应中回显或暴露密码相关信息。

**Rationale**

- BCrypt 是 Web 行业实际应用最广泛的密码哈希算法之一，具备抗暴力破解能力，并支持内置的工作因子（cost）调整；
- Spring 生态中对 BCrypt 有成熟支持（如 `BCryptPasswordEncoder`），可以降低实现复杂度；
- 与“只用 MD5/SHA1”等简易哈希相比，BCrypt 对 GPU/ASIC 攻击更友好，符合通用安全实践。

**Alternatives considered**

- 明文存储或简单可逆加密：严重违反安全基线，不可接受；
- 自定义密码加密算法：缺乏业界审计，维护成本高，且仍需解决哈希参数与升级策略问题；
- 使用 PBKDF2 / Argon2：安全性同样优秀，但考虑到项目体量与 Spring 生态的直接支持，BCrypt 足以满足当前需求。

## 决策 3：账号标识模型（手机号 / 邮箱）

**Decision**

- 用户账号支持使用“手机号”或“邮箱”作为登录标识：
  - 数据模型中为用户实体提供 `phone` 与 `email` 两个字段，允许只填其一，但至少保证注册时有一个登录标识存在；
  - `phone` 与 `email` 分别在数据库层面设置唯一约束；
  - 登录请求中，客户端需显式指定使用哪种标识（例如前端表单中将手机号与邮箱区分），后端根据字段选择进行查询。

**Rationale**

- 规格中明确提到“手机号或邮箱”登录，双标识模型更贴近实际业务；
- 将两种标识分列字段、分别唯一，可以在后续扩展（如手机/邮箱绑定变更）时更灵活；
- 登录时显式区分，有利于错误信息控制（避免过度暴露“账号是否存在”）。

**Alternatives considered**

- 使用单一 `username` 字段混合存储手机号或邮箱：实现简单，但在数据层难以做细粒度的唯一约束，后续做账号绑定/分渠道统计会较困难；
- 强制只支持手机号或只支持邮箱：与当前需求不符，降低用户体验。

## 决策 4：登录失败防护策略

**Decision**

- 实现基础的“短时间多次失败限制”机制，减少暴力破解风险：
  - 在后端维护基于“账号 + 客户端维度（如 IP 或设备标识）”的失败计数；
  - 当连续失败次数超过阈值（例如 5 次）时，在一段时间内（例如 15 分钟）临时禁止该组合继续尝试登录；
  - 失败计数与封禁状态可以先采用内存缓存方案实现（如基于本进程内的 Map 或框架缓存），后续视需要扩展到 Redis 等集中式存储。

**Rationale**

- 满足规格中对“基础防护”的要求，不引入额外复杂基础设施的前提下即可生效；
- 结合 Sa-Token 的会话管理，防护策略只作用于登录接口，对业务接口影响有限；
- 初期用户规模较小，单机内存级计数方案足够，未来如需多节点水平扩展再平滑升级到集中缓存/Redis 方案。

**Alternatives considered**

- 完全依赖 Web 服务器或 WAF 的限流配置：对应用层而言不可见，不便于针对账号级规则做细粒度调整；
- 直接接入 Redis + 滑动窗口算法：安全性最佳，但在当前项目阶段会引入新的基础组件，增加运维与开发复杂度。

## 决策 5：会话与过期策略

**Decision**

- 会话有效性策略：
  - 登录会话默认设置“空闲超时”窗口（例如 2 小时未操作则失效）；
  - 可选地设置“最大存活时长”（例如 7 天内无论是否操作都要求重新登录），具体取值在配置中管理；
  - 允许同一账号多终端同时在线（多端登录），后续如有需求再引入“单端登录”等策略；
  - 会话采用 Sa-Token 与 JWT 的集成模式管理 token 形式，由配置决定是普通 token 还是 JWT token，但在业务与前端契约中只体现为“token 字符串”，不暴露实现细节。

**Rationale**

- 2 小时空闲超时时间兼顾安全与体验，常见于 Web 应用；
- 允许多端登录更符合知识库类应用的使用习惯（PC + 手机）；
- 在规范层和接口契约中抽象为“token”而非具体实现细节，有利于未来替换 token 形式或加密策略。

**Alternatives considered**

- 非常短的超时时间（如 15 分钟）：安全性略高，但对写作/阅读类场景用户体验不佳；
- 强制单端登录：对安全要求极高的系统有效，但会影响多终端使用场景，与当前产品定位不完全匹配。

## 决策 6：日志与审计策略

**Decision**

- 对以下关键事件进行统一审计记录：
  - 登录成功（含账号标识、设备/客户端信息、IP、大致时间）；
  - 登录失败（包括失败原因分类，如密码错误、账号状态异常、多次失败被限制等）；
  - 退出登录（包括触发方式：用户主动、会话超时等）。
- 日志保留时间应满足规格中“可回溯近 30 天关键登录行为”的要求，具体的日志落盘与归档策略依赖整体运维方案。

**Rationale**

- 便于安全事件追踪与问题排查，满足成功标准 SC-004 的约束；
- 在实现上可以复用现有日志框架（如基于 SLF4J/Logback），不强制引入额外组件。

**Alternatives considered**

- 不记录失败登录或退出事件：虽然实现简单，但在发生安全事件时无法追踪攻击来源与行为模式，不符合安全实践。
