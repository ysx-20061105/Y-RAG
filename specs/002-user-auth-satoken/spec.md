# Feature Specification: 用户登录注册与认证

**Feature Branch**: `002-user-auth-satoken`
**Created**: 2026-03-30
**Status**: Draft
**Input**: User description: "完成用户登录注册功能接口，认证使用satoken，https://sa-token.cc/v/v1.9.0/doc/index.html#/"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用户注册账号 (Priority: P1)

作为新用户，我可以通过手机号或邮箱注册账号并设置登录密码，这样我才能创建属于自己的知识库与笔记。

**Why this priority**: 没有账号体系，就无法进行后续的知识库管理和笔记 CRUD，是整个系统的入口能力，必须优先完成。

**Independent Test**:
- 调用注册接口完成一次成功注册；
- 验证数据库中生成唯一用户记录；
- 使用新账号能顺利登录并访问受保护接口。

**Acceptance Scenarios**:

1. **Given** 注册信息（手机号/邮箱未被使用，密码符合复杂度要求），**When** 用户提交注册请求，**Then** 系统创建新用户并返回注册成功结果。
2. **Given** 注册信息（手机号/邮箱已存在），**When** 用户提交注册请求，**Then** 系统拒绝注册并提示“账号已存在”。
3. **Given** 注册信息不完整或不合法（如缺少必填字段、格式错误），**When** 用户提交注册请求，**Then** 系统返回参数校验错误信息。

---

### User Story 2 - 用户登录获取会话 (Priority: P1)

作为已注册用户，我可以通过账号密码登录系统，系统基于 Sa-Token 创建登录会话，让我在一段时间内可以访问自己的知识库和笔记数据，而无需每次都重新输入密码。

**Why this priority**: 登录能力是所有受保护接口的前置条件，直接影响整体可用性和安全性，与注册同为 P1。

**Independent Test**:
- 使用已注册账号调用登录接口；
- 验证返回登录态凭证（如 Sa-Token token）；
- 携带凭证访问一个受保护接口，返回成功；
- 不携带或携带无效凭证访问时返回未登录/无权限。

**Acceptance Scenarios**:

1. **Given** 有效账号密码，**When** 用户提交登录请求，**Then** 系统验证通过并创建登录会话，返回登录成功及相关会话信息（如 token）。
2. **Given** 错误密码或不存在的账号，**When** 用户提交登录请求，**Then** 系统拒绝登录并返回明确错误提示，不创建会话。
3. **Given** 用户多次连续输错密码，**When** 用户继续尝试登录，**Then** 系统可以触发保护机制（如短期内限制尝试频率），以降低暴力破解风险。

---

### User Story 3 - 用户退出登录与会话失效 (Priority: P2)

作为已登录用户，我希望可以主动退出登录或在会话超时后自动失效，这样可以降低账号在公共设备上的被滥用风险。

**Why this priority**: 虽然不影响基础使用，但与安全密切相关，有助于保护用户隐私与数据安全。

**Independent Test**:
- 已登录状态下调用登出接口，随后访问受保护接口应返回未登录；
- 将会话模拟到期后访问受保护接口，应提示重新登录。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 用户调用退出登录接口，**Then** 当前会话被立即注销，后续携带原 token 的请求都被视为未登录。
2. **Given** 用户长时间未进行任何操作，**When** 会话超出预设的空闲时长，**Then** 会话自动失效，用户再次访问受保护资源时被要求重新登录。

---

### Edge Cases

- 注册时并发提交相同手机号/邮箱：系统应确保最终只创建一个用户记录，其余请求返回“账号已存在”。
- 登录时频繁尝试错误密码：系统应限制短时间内的失败次数，避免被暴力破解。
- 退出登录后浏览器仍保留旧 token：携带旧 token 的请求必须被视为未登录，不能访问受保护资源。
- 使用过期或伪造的 token 访问接口：系统应统一返回未登录/会话无效的错误提示，并记录安全日志。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供用户注册接口，支持通过手机号或邮箱+密码创建新账号，并确保账号唯一。
- **FR-002**: 系统必须对注册信息进行格式与完整性校验，包括账号格式、密码长度与复杂度等，不符合规则时返回明确错误信息。
- **FR-003**: 系统必须提供用户登录接口，支持用户通过账号（手机号或邮箱）+密码登录，并在验证通过后建立登录会话。
- **FR-004**: 系统必须基于认证框架维护登录态，会话在有效期内允许用户访问受保护的业务接口，在会话失效或注销后拒绝访问。
- **FR-005**: 系统必须提供退出登录接口，用户调用后当前登录会话立即失效。
- **FR-006**: 系统必须对所有受保护的后端接口进行登录态校验，未登录或会话无效时统一返回未认证错误。
- **FR-007**: 系统应支持配置会话有效期和空闲超时时长，到期后自动使会话失效，防止长期占用。
- **FR-008**: 系统应对多次登录失败实施基础防护（如短时间内失败次数限制或延迟响应），降低暴力破解风险。
- **FR-009**: 系统应记录关键登录事件（成功登录、失败登录、退出登录），以便后续审计与问题排查。

### Key Entities *(include if feature involves data)*

- **用户账号（User）**：代表系统中的终端用户身份，包含账号标识（手机号或邮箱）、密码凭证（安全存储）、状态（正常/禁用）、创建与更新时间等信息，是登录注册功能的核心实体。
- **登录会话（Session/Token）**：代表用户当前登录状态的抽象，包含与用户关联的会话标识、创建时间、过期时间、状态（有效/失效）等信息，用于在后续请求中识别用户身份并控制访问权限。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户能够在一次无错误操作内完成注册+登录流程的比例不低于 90%。
- **SC-002**: 在正常网络条件下，从用户提交登录请求到看到登录成功反馈的时间不超过 2 秒（后端处理时间层面）。
- **SC-003**: 在约定的时间窗口内，因认证问题（未登录、会话过期、密码错误）产生的工单或反馈占总体问题反馈的比例不高于 20%。
- **SC-004**: 通过集中日志或审计信息可以还原至少最近 30 天内的关键登录行为（成功登录、频繁失败、退出登录），用于安全排查。

## Assumptions

- 系统面向的用户具备基本的上网环境和终端设备，能够通过 Web 或前端应用访问后端接口。
- 首个版本仅支持通过手机号或邮箱 + 密码的方式登录，第三方登录和多因素认证暂不在本次范围内。
- 密码复杂度和会话时长采用符合常规 Web 系统安全实践的默认配置，如最小长度、字符种类要求和合理的超时时间。
- 现有或后续将存在统一的用户中心/权限体系，本次功能只聚焦于基础登录注册与会话管理，不涉及复杂的角色与细粒度授权规则。
