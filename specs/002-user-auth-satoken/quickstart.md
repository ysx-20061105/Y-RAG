# Quickstart: 用户登录注册与认证

本快速上手指南面向调用后端接口的前端或其他客户端开发者，说明如何使用本项目提供的注册、登录与退出登录接口，以及如何在后续请求中携带认证凭证。

> 说明：接口的精确字段与错误码以 `contracts/` 目录中的契约文档为准，本指南聚焦“怎么用”和典型交互流程。

---

## 1. 基础约定

- 接口前缀：`/api/v1/auth`（示例，不排除后续调整）。
- 传输格式：HTTP + JSON。
- 认证方式：基于 Sa-Token 的 token（下文统称为“token”），通常通过 HTTP Header 传递，例如：
  - `Authorization: Bearer <token>` 或
  - 自定义 Header（以实际契约为准）。

在阅读以下示例前，请确保：

- 已获取后端服务的基准 URL，例如：`https://api.example.com` 或本地开发地址 `http://localhost:8080`。

---

## 2. 用户注册

### 2.1 场景说明

新用户通过手机号或邮箱 + 密码创建账号。注册成功后，通常前端会引导用户进入登录流程。

### 2.2 请求示例

```http
POST /api/v1/auth/register
Content-Type: application/json

{
  "phone": "13800138000",        // 可选，与 email 至少填一个
  "email": "user@example.com",   // 可选，与 phone 至少填一个
  "password": "P@ssw0rd123"      // 必填，需满足密码复杂度要求
}
```

### 2.3 响应示例

```json
{
  "success": true,
  "data": {
    "userId": "1234567890"       // 系统内部的用户标识
  }
}
```

可能的错误情况包括（但不限于）：

- 注册信息不完整或格式非法（例如手机号/邮箱格式错误、密码太弱）。
- 手机号或邮箱已被其他账号使用。

错误响应将通过统一的 ApiResponse 格式返回，前端需根据 `success` 字段和错误信息进行处理。

---

## 3. 用户登录

### 3.1 场景说明

已注册用户通过账号（手机号或邮箱）+ 密码登录，后端在验证通过后创建会话并返回 token。前端需要在后续请求中携带该 token 以访问受保护接口。

### 3.2 按手机号登录示例

```http
POST /api/v1/auth/login
Content-Type: application/json

{
  "identifierType": "PHONE",      // 登录标识类型：PHONE 或 EMAIL
  "identifier": "13800138000",   // 对应的手机号或邮箱
  "password": "P@ssw0rd123"      // 用户密码
}
```

### 3.3 按邮箱登录示例

```http
POST /api/v1/auth/login
Content-Type: application/json

{
  "identifierType": "EMAIL",
  "identifier": "user@example.com",
  "password": "P@ssw0rd123"
}
```

### 3.4 成功响应示例

```json
{
  "success": true,
  "data": {
    "userId": "1234567890",
    "token": "xxxxx.yyyyy.zzzzz",   // 具体格式由后端实现决定
    "expiresIn": 7200                // 可选，单位秒，表示预计有效期
  }
}
```

前端应：

1. 将 token 安全地保存（例如内存、受控的持久化存储），避免通过不安全渠道泄露；
2. 在访问受保护接口时，按约定的方式携带 token（见下一节）；
3. 在收到会话失效类错误时，引导用户重新登录。

### 3.5 登录失败与防暴力破解

当用户输入错误密码、账号不存在或账号状态异常时，后端会返回统一的错误响应。对于短时间内多次失败的组合（账号 + 客户端），后端可能触发临时限制（如锁定 15 分钟）。

前端可根据约定的错误码或错误信息进行适当提示，例如：

- 普通错误：提示“账号或密码错误”；
- 被限制：提示“尝试次数过多，请稍后再试”。

---

## 4. 携带 token 访问受保护接口

### 4.1 场景说明

登录成功后，用户可以访问需要登录态的业务接口（例如笔记 CRUD、知识库管理等）。所有此类接口会统一校验 token。

### 4.2 请求示例

假设某受保护接口为：`GET /api/v1/notes`，示例请求：

```http
GET /api/v1/notes
Authorization: Bearer xxxxx.yyyyy.zzzzz
```

或（如果契约使用自定义 Header）：

```http
GET /api/v1/notes
X-Auth-Token: xxxxx.yyyyy.zzzzz
```

> 实际使用哪种 Header 名称与格式，以 `contracts/` 中约定为准。

### 4.3 未登录或会话失效时的行为

当：

- 未携带 token；
- 携带的 token 无效（伪造、格式错误等）；
- token 对应的会话已过期或已被注销；

后端会统一返回“未登录/会话无效”类错误，前端可据此：

- 跳转到登录页；
- 弹出提示，引导用户重新登录；
- 清除本地缓存的旧 token。

---

## 5. 退出登录

### 5.1 场景说明

用户希望在当前设备/浏览器上退出登录，以防止他人继续访问其数据。退出登录应立即使当前 token 失效。

### 5.2 请求示例

```http
POST /api/v1/auth/logout
Authorization: Bearer xxxxx.yyyyy.zzzzz
```

### 5.3 响应示例

```json
{
  "success": true
}
```

退出登录后：

- 前端应立即清除本地存储的 token；
- 使用旧 token 调用任何受保护接口都将被视为未登录。

---

## 6. 会话过期与自动重新登录

### 6.1 会话过期场景

根据后端配置，会话可能在以下情况下失效：

- 长时间未操作（超过空闲超时时间，例如 2 小时）；
- 超过最大存活时长（例如 7 天）。

当会话过期时：

- 受保护接口会返回统一的“未登录/会话无效”错误；
- 前端应清理本地 token，并提示用户重新登录。

### 6.2 前端推荐处理方式

- 在全局请求拦截器中统一处理“未登录/会话无效”错误码；
- 自动跳转到登录页，或弹出对话框要求用户重新登录；
- 可在登录成功后回跳到用户之前访问的页面，以优化体验。

---

## 7. 本地开发与调试建议

- 在本地环境下，可使用 `http://localhost:8080` 等地址进行接口调试；
- 推荐使用 Postman、curl 或浏览器开发者工具手动验证以下步骤：
  1. 注册一个新账号；
  2. 使用该账号登录并获得 token；
  3. 携带 token 访问一个受保护接口，确认能正常返回数据；
  4. 调用退出登录接口，再次访问受保护接口，确认被视为未登录；
  5. 等待会话过期或模拟过期后，确认访问受保护接口时需要重新登录。

通过上述步骤，前端开发者可以快速验证后端认证链路是否可用，并据此集成到实际应用中。
