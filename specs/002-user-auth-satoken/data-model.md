# Data Model: 用户登录注册与认证

本文件基于 `spec.md` 与 `research.md`，描述本功能涉及的核心数据模型及其业务规则。仅关注“是什么”和“规则”，不绑定具体实现细节（如 Java 类、表结构）。

---

## 1. 核心实体：User（用户账号）

### 1.1 概述

User 实体表示系统中的终端用户账号，是登录、会话管理以及后续权限扩展的基础。每个 User 代表一个独立的用户身份。

### 1.2 字段与含义

- **id**
  - 含义：用户的全局唯一标识符。
  - 规则：
    - 系统内部生成，用户不可自行指定或修改。
    - 在系统中唯一，不与其他实体混用。

- **phone**
  - 含义：用户的手机号登录标识。
  - 规则：
    - 可为空；当为空时，用户只能使用 email 登录。
    - 若不为空，必须符合手机号格式（根据目标市场约定，例如中国大陆 11 位手机号规则）。
    - 在系统中全局唯一（不存在两个用户拥有相同的非空手机号）。

- **email**
  - 含义：用户的邮箱登录标识。
  - 规则：
    - 可为空；当为空时，用户只能使用 phone 登录。
    - 若不为空，必须符合邮箱格式。
    - 在系统中全局唯一（不存在两个用户拥有相同的非空邮箱）。

- **passwordHash**
  - 含义：用户密码的安全哈希值。
  - 规则：
    - 只存储基于 BCrypt 的单向哈希值，不存储明文密码，也不存储可逆加密结果。
    - 注册与修改密码时，必须使用当前约定的 BCrypt 参数（例如 cost 因子）进行计算。
    - 任何接口均不得回传该字段或与之等价的敏感信息。

- **status**
  - 含义：账号状态。
  - 规则：
    - 典型取值：`NORMAL`（正常）、`DISABLED`（禁用 / 封禁）。
    - 处于 `DISABLED` 状态的账号不得完成登录，即使密码正确也应返回“账号状态异常”类错误。
    - 后续如需扩展更多状态（如待激活、已删除），需在此基础上演进。

- **createdAt**
  - 含义：账号创建时间。
  - 规则：
    - 由系统在创建用户时填写，不可被外部直接修改。

- **updatedAt**
  - 含义：账号最近一次被修改的时间。
  - 规则：
    - 在用户关键信息（如密码、手机号、邮箱、状态）发生变更时更新。

### 1.3 字段约束与业务规则

- 至少一个登录标识存在：
  - 注册时必须提供 `phone` 或 `email` 中至少一个，二者同时为空视为无效注册请求。
  - 允许仅提供其中一个，后续可通过绑定流程补充另一个。

- 唯一性与并发：
  - `phone` 与 `email` 在系统中各自全局唯一。
  - 在高并发注册场景下，系统应确保最终只产生一条成功记录，其余因冲突导致的注册应返回“账号已存在”类错误。

- 密码策略：
  - passwordHash 对应的原始密码需满足最低复杂度要求（例如最小长度、字符种类等，具体规则在接口契约及校验逻辑中体现）。
  - 修改密码时必须重新计算并覆盖 passwordHash。

## 2. 辅助模型：登录失败计数（LoginFailureCounter）

### 2.1 概述

为支持基础的暴力破解防护，本功能需要对“账号 + 客户端维度”组合维护短期的登录失败次数与封禁信息。该模型可以通过内存缓存、集中缓存（如 Redis）或数据库实现，此处仅描述其语义与规则。

### 2.2 标识与字段

- **key**
  - 含义：表示某一“账号 + 客户端维度”的组合标识。
  - 示例：可以由登录时使用的账号标识（phone 或 email）与客户端 IP 或设备标识组合计算得到的字符串。
  - 规则：
    - 对同一账号和同一客户端，key 值应一致，以便累计失败次数。

- **failureCount**
  - 含义：在当前时间窗口内的连续登录失败次数。
  - 规则：
    - 每次登录失败时 +1。
    - 登录成功时清零并重置相关封禁状态。
    - 当超过预设阈值（例如 5 次）时触发临时封禁。

- **blockedUntil**
  - 含义：当前组合被临时禁止尝试登录的截止时间。
  - 规则：
    - 当 failureCount 超过阈值时，将 blockedUntil 设置为“当前时间 + 预设封禁时长”（例如 15 分钟）。
    - 在 blockedUntil 之前的任何登录尝试都应直接拒绝，并返回合适的错误信息，而无需再验证密码。

### 2.3 生命周期与清理

- 对于长期无活动的 key（例如在一定时间窗外不再有登录尝试），系统可以按实现策略自动清理，以避免无限制增长。
- 初始阶段可将该信息保存在应用内存中，后续如需多节点扩展，可以迁移到集中缓存方案中，语义不变。

## 3. 会话与 Token 抽象

会话与 Token 的底层实现由 Sa-Token 提供，此处只抽象出与业务和前端契约相关的部分。

### 3.1 会话抽象字段

- **sessionId / token**
  - 含义：代表当前登录会话的唯一标识字符串。
  - 规则：
    - 登录成功后由认证框架生成并返回给客户端。
    - 客户端在后续请求中通过约定方式（如 HTTP Header）携带。

- **userId**
  - 含义：当前会话绑定的用户标识，对应 User.id。

- **createdAt**
  - 含义：会话创建时间。

- **expiresAt**（可选）
  - 含义：会话最大存活时间点。
  - 规则：
    - 由配置的“最大存活时长”推导。

- **lastAccessAt**（可选）
  - 含义：会话最近一次访问受保护资源的时间。
  - 规则：
    - 用于支持“空闲过期”策略（例如 2 小时未操作则失效）。

### 3.2 会话规则

- 会话有效性：
  - 在未超出最大存活时长且未超过空闲超时时间的前提下，会话视为有效。
  - 超出任一限制后，会话视为失效，受保护接口需要返回未登录/会话无效的错误。

- 多端登录：
  - 允许同一 User 同时存在多个会话（多终端登录），具体数量上限可以通过配置控制。

- 登出行为：
  - 用户主动调用退出登录接口时，应立即将当前 token 对应的会话标记为失效，后续携带该 token 的请求一律按未登录处理。

## 4. 日志与审计事件模型

为满足审计和安全排查需求，需要定义与登录相关的关键事件模型。实际落地可使用日志系统或审计表，此处只定义事件内容。

### 4.1 登录成功事件（LoginSuccessEvent）

- **userId**：成功登录的用户标识。
- **identifierType**：使用的账号标识类型（`PHONE` 或 `EMAIL`）。
- **identifierMasked**：脱敏后的手机号或邮箱（如中间部分用 `*` 代替）。
- **clientIp**：客户端 IP 信息。
- **userAgent**（可选）：客户端 User-Agent 信息。
- **timestamp**：事件发生时间。

### 4.2 登录失败事件（LoginFailureEvent）

- **identifierType**：`PHONE` 或 `EMAIL`。
- **identifierMasked**：脱敏后的账号标识。
- **clientIp**：客户端 IP。
- **reason**：失败原因分类（如 `BAD_CREDENTIALS`、`ACCOUNT_STATUS_INVALID`、`BLOCKED_BY_BRUTE_FORCE_PROTECTION` 等）。
- **timestamp**：事件发生时间。

> 说明：登录失败事件不绑定 userId，以避免在“账号不存在”场景下暴露过多信息。

### 4.3 退出登录事件（LogoutEvent）

- **userId**：退出登录的用户标识。
- **clientIp**：客户端 IP。
- **triggerType**：触发方式（如 `USER主动`, `SESSION_TIMEOUT` 等）。
- **timestamp**：事件发生时间。

### 4.4 审计规则

- 至少保留最近 30 天内的关键登录行为记录，以满足 SC-004 要求。
- 日志中不得记录明文密码或可用于直接重放的敏感凭证。

## 5. 与后续设计的关系

- 本数据模型将指导：
  - 接口契约中字段命名与校验逻辑（例如注册/登录请求和响应体结构）。
  - 后续数据库表结构与索引设计（如对 phone/email 的唯一约束）。
  - 登录失败防护与会话管理实现中使用的键与状态结构。
- 如未来引入更复杂的权限模型（如角色、权限点），可以在 User 基础上增加关联实体，而无需改变本文件已定义的核心语义。
